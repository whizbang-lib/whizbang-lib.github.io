# Troubleshooting Guide

Comprehensive **troubleshooting guide** for Whizbang applications - common issues, debugging techniques, diagnostic tools, and step-by-step solutions.

---

## Common Issues

| Issue | Symptom | Root Cause | Solution |
|-------|---------|------------|----------|
| **No Handler Found** | `InvalidOperationException` | Generated registrations not called | Call `AddWhizbangDispatcher()` + `AddReceptors()` |
| **Message Not Published** | Events not reaching subscribers | Outbox workers not draining | Check `AddWhizbang()` wiring + `wh_outbox` backlog |
| **Duplicate Processing** | Side effects run twice | Non-idempotent receptor retried | Use `AttemptNumber` / `[ReceptorIdempotent]` |
| **High Latency** | Slow responses | Database query | Add indexes |
| **Memory Leak** | Memory grows indefinitely | Unclosed connections | Use `await using` |

## Issue 1: No Handler Found

### Symptom

```
System.InvalidOperationException: No handler registered for CreateOrderCommand
```

### Diagnosis

```csharp{title="Diagnosis" description="Diagnosis" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Diagnosis"]}
// Check DI container
var dispatcher = serviceProvider.GetService<IDispatcher>();
var receptor = serviceProvider.GetService<IReceptor<CreateOrderCommand, OrderCreatedEvent>>();

if (receptor is null) {
  Console.WriteLine("ERROR: Receptor not registered!");
}

// Or ask the registry directly
var registry = serviceProvider.GetRequiredService<IReceptorRegistry>();
var receptors = registry.GetReceptorsFor(typeof(CreateOrderCommand), LifecycleStage.LocalImmediateInline);
Console.WriteLine($"Registered receptors: {receptors.Count}");
```

### Solution: Call the Generated Registration Extensions

Receptor discovery is source-generated - **don't** register `GeneratedDispatcher` or individual receptors by hand. Make sure the generated extensions run at startup:

```csharp{title="Solution: Use Source-Generated Registration" description="Generated DispatcherRegistrations extensions register everything" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Source-Generated"] unverified="DI registration wiring — AddWhizbangDispatcher()/AddReceptors() call generated extensions; configuration, not a mapped-test behavior"}
// ✅ Generated by Whizbang.Generators (DispatcherRegistrations class in your assembly)
builder.Services.AddWhizbangDispatcher();  // GeneratedDispatcher + IReceptorRegistry + IReceptorInvoker
builder.Services.AddReceptors();           // registers every discovered receptor
```

If the extensions don't exist, the generator didn't run - verify the `Whizbang.Generators` package is referenced by the project that *contains* the receptors, and that the receptor implements `IReceptor<TMessage>` / `IReceptor<TMessage, TResponse>`.

## Issue 2: Message Not Published

### Symptom

Events not reaching subscribers (no errors in logs).

### Diagnosis

```sql{title="Diagnosis (2)" description="Diagnosis (2)" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Diagnosis"]}
-- Check Whizbang outbox backlog (completed rows are deleted, so this is pending work)
SELECT COUNT(*) FROM wh_outbox WHERE processed_at IS NULL;

-- If the count keeps growing, the outbox workers aren't draining
```

The `whizbang.queue.estimated_depth{queue_name="outbox"}` gauge tracks the same number continuously.

### Solution 1: Whizbang Workers Not Wired

**Problem**: The publishing pipeline (claim/drain/publish workers) isn't running. The workers are registered by `AddWhizbang()` (which calls `AddWhizbangWorkers()` internally) together with a storage driver.

**Fix**:

```csharp{title="Solution 1: Whizbang Workers Not Wired" description="AddWhizbang + storage driver registers the outbox pipeline" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Outbox"] unverified="DI/driver wiring — AddWhizbang().WithEFCore().WithDriver.Postgres registration; configuration, not a mapped-test behavior"}
builder.Services
  .AddWhizbang()               // registers workers (claim, drain, publish, heartbeat)
  .WithEFCore<MyDbContext>()
  .WithDriver.Postgres;        // wires IWorkCoordinator for the driver
```

Don't add Whizbang worker classes with `AddHostedService<...>()` yourself - duplicate registrations cause competing pollers.

### Solution 2: Transport Not Ready / Subscriptions Failed

**Problem**: Messages are claimed but publishing fails because the transport is down or misconfigured.

**Diagnosis**: check the built-in `subscriptions` health check (reports `Degraded`/`Unhealthy` with the failed destinations in its data) and transport connectivity checks (`azure_servicebus`, `rabbitmq`). Raise trace verbosity for outbox visibility:

```csharp{title="Solution 2: Transport Diagnosis" description="Trace the outbox path" category="Configuration" difficulty="INTERMEDIATE" tags=["Operations", "Deployment", "Solution", "Transport"] unverified="tracing configuration — sets AddWhizbang tracing options; configuration, not a mapped-test behavior"}
services.AddWhizbang(options => {
  options.Tracing.Verbosity = TraceVerbosity.Verbose;      // includes outbox/inbox operations
  options.Tracing.Components = TraceComponents.Messaging;  // Messages | Events | Outbox | Inbox
});
```

### Solution 3: Service Bus Connection String Wrong

**Problem**: Wrong connection string or permissions.

**Diagnosis**:

```bash{title="Solution 3: Service Bus Connection String Wrong" description="Solution 3: Service Bus Connection String Wrong" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Service"]}
# Test Service Bus connection
curl -X POST https://myservicebus.servicebus.windows.net/orders/messages \
  -H "Authorization: SharedAccessSignature ..." \
  -d '{"test": true}'
```

**Fix**:

```json{title="Solution 3: Service Bus Connection String Wrong" description="Solution 3: Service Bus Connection String Wrong" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Service"]}
{
  "AzureServiceBus": {
    "ConnectionString": "Endpoint=sb://myservicebus.servicebus.windows.net/;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=***"
  }
}
```

## Issue 3: Duplicate Processing

### Symptom

Message handled twice (duplicate database inserts).

### Diagnosis

Inbox deduplication is **built into Whizbang** - the inbox rejects messages whose `MessageId` was already recorded, and the `whizbang.dispatcher.duplicates_detected` counter increments for each rejection. If you still see double side effects:

```sql{title="Diagnosis (3)" description="Diagnosis (3)" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Diagnosis"]}
-- Whizbang's inbox table - completed rows are deleted, in-flight rows are visible
SELECT message_id, received_at FROM wh_inbox ORDER BY received_at DESC LIMIT 50;
```

### Solution 1: Side Effects Outside the Dedup Boundary

**Problem**: dedup protects the *message pipeline*, not arbitrary side effects. A receptor that fires on a **Detached** lifecycle stage or performs non-idempotent external calls can still run twice if perspective processing succeeds but the checkpoint save fails and the work item retries.

**Fix**: make retry-sensitive receptors idempotent, or check the attempt number from the lifecycle context:

```csharp{title="Solution 1: Retry-Aware Receptor" description="Skip retries using ILifecycleContext.AttemptNumber" category="Configuration" difficulty="INTERMEDIATE" tags=["Operations", "Deployment", "Solution", "Inbox"] unverified="user-authored receptor illustrating idempotency via ILifecycleContext.AttemptNumber; not asserted by a mapped unit test"}
public ValueTask HandleAsync(OrderCreatedEvent @event, CancellationToken cancellationToken = default) {
  if (_lifecycleContext?.AttemptNumber > 1) {
    return ValueTask.CompletedTask;  // already fired on the first attempt
  }

  // ... non-idempotent side effect (send email, call external API)
  return ValueTask.CompletedTask;
}
```

During replays/rebuilds, non-idempotent receptors are skipped automatically for already-processed events; opt in deliberately with `[ReceptorIdempotent(AlwaysFire = true)]` and branch on `ILifecycleContext.IsReplay`.

### Solution 2: Service Bus PeekLock Not Used

**Problem**: Using `ReceiveAndDelete` mode (at-most-once delivery).

**Fix**: Use `PeekLock` mode (at-least-once delivery):

```csharp{title="Solution 2: Service Bus PeekLock Not Used" description="Fix: Use PeekLock mode (at-least-once delivery):" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Service"] unverified="Azure Service Bus SDK configuration (PeekLock ReceiveMode); external component, not a Whizbang behavior"}
var processor = client.CreateProcessor("orders", new ServiceBusProcessorOptions {
  ReceiveMode = ServiceBusReceiveMode.PeekLock,  // ✅ Use PeekLock
  MaxConcurrentCalls = 10
});
```

## Issue 4: High Latency

### Symptom

API responses taking > 1 second.

### Diagnosis

**Application Insights query**:

```kusto
requests
| where timestamp > ago(1h)
| summarize p95 = percentile(duration, 95) by name
| where p95 > 1000  // > 1 second
| order by p95 desc
```

**PostgreSQL slow query log**:

```sql{title="Diagnosis (4)" description="PostgreSQL slow query log:" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Diagnosis"]}
-- Enable slow query log
ALTER DATABASE orders SET log_min_duration_statement = 1000;  -- Log queries > 1s

-- View slow queries
SELECT query, mean_exec_time, calls
FROM pg_stat_statements
WHERE mean_exec_time > 1000
ORDER BY mean_exec_time DESC
LIMIT 10;
```

### Solution 1: Missing Index

**Problem**: Full table scan on large table.

**Diagnosis**:

```sql{title="Solution 1: Missing Index" description="Solution 1: Missing Index" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Missing"]}
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 'cust-123';

-- Output:
-- Seq Scan on orders (cost=0.00..100000.00 rows=1 width=100)
-- Planning Time: 0.5 ms
-- Execution Time: 1234.5 ms
```

**Fix**: Add index:

```sql{title="Solution 1: Missing Index (2)" description="Fix: Add index:" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Missing"]}
CREATE INDEX idx_orders_customer_id ON orders(customer_id);

-- After index:
-- Index Scan using idx_orders_customer_id (cost=0.29..8.31 rows=1 width=100)
-- Planning Time: 0.5 ms
-- Execution Time: 1.2 ms
```

### Solution 2: N+1 Query Problem

**Problem**:

```csharp{title="Solution 2: N+1 Query Problem" description="Solution 2: N+1 Query Problem" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "N+1"] unverified="counter-example — intentionally inefficient N+1 query, nothing to assert"}
// ❌ N+1 queries (1 for orders, N for items)
var orders = await _db.QueryAsync<OrderRow>("SELECT * FROM orders WHERE customer_id = @CustomerId");

foreach (var order in orders) {
  order.Items = await _db.QueryAsync<OrderItemRow>(
    "SELECT * FROM order_items WHERE order_id = @OrderId",
    new { OrderId = order.OrderId }
  );
}
```

**Fix**: Use JOIN or batch query:

```csharp{title="Solution 2: N+1 Query Problem (2)" description="Fix: Use JOIN or batch query:" category="Configuration" difficulty="INTERMEDIATE" tags=["Operations", "Deployment", "Solution", "N+1"] unverified="user application query code (Dapper-style JOIN); not a Whizbang API"}
// ✅ Single query with JOIN
var orders = await _db.QueryAsync<OrderRow, OrderItemRow, OrderRow>(
  """
  SELECT o.*, i.*
  FROM orders o
  LEFT JOIN order_items i ON o.order_id = i.order_id
  WHERE o.customer_id = @CustomerId
  """,
  (order, item) => {
    order.Items.Add(item);
    return order;
  },
  new { CustomerId = customerId }
);
```

### Solution 1: Unclosed Connections

**Problem**:

```csharp{title="Solution 1: Unclosed Connections" description="Solution 1: Unclosed Connections" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Unclosed"] unverified="counter-example — intentionally leaks an undisposed connection, nothing to assert"}
// ❌ Connection not disposed
var connection = new NpgsqlConnection(connectionString);
await connection.OpenAsync();

// Process data...

// Missing: await connection.DisposeAsync();
```

**Fix**:

```csharp{title="Solution 1: Unclosed Connections (2)" description="Solution 1: Unclosed Connections" category="Configuration" difficulty="BEGINNER" tags=["Operations", "Deployment", "Solution", "Unclosed"] unverified="general await-using disposal pattern (Npgsql); not a Whizbang API"}
// ✅ Use await using
await using var connection = new NpgsqlConnection(connectionString);
await connection.OpenAsync();

// Process data...

// Automatically disposed
```

### Solution 2: Event Handler Not Unsubscribed

**Problem**:

```csharp{title="Solution 2: Event Handler Not Unsubscribed" description="Solution 2: Event Handler Not Unsubscribed" category="Configuration" difficulty="INTERMEDIATE" tags=["Operations", "Deployment", "Solution", "Event"] unverified="counter-example — intentionally leaks via an unsubscribed event handler, nothing to assert"}
// ❌ Event handler keeps object alive
public class OrderProcessor {
  public OrderProcessor(IEventBus eventBus) {
    eventBus.OrderCreated += HandleOrderCreated;  // Never unsubscribed
  }

  private void HandleOrderCreated(OrderCreated @event) {
    // Process order...
  }
}
```

**Fix**:

```csharp{title="Solution 2: Event Handler Not Unsubscribed - OrderProcessor" description="Solution 2: Event Handler Not Unsubscribed - OrderProcessor" category="Configuration" difficulty="INTERMEDIATE" tags=["Operations", "Deployment", "Solution", "Event"] unverified="general IDisposable event-unsubscribe pattern; not a Whizbang API"}
// ✅ Unsubscribe when disposed
public class OrderProcessor : IDisposable {
  private readonly IEventBus _eventBus;

  public OrderProcessor(IEventBus eventBus) {
    _eventBus = eventBus;
    _eventBus.OrderCreated += HandleOrderCreated;
  }

  public void Dispose() {
    _eventBus.OrderCreated -= HandleOrderCreated;  // ✅ Unsubscribe
  }
}
```

### 2. Use Correlation IDs

```csharp{title="Use Correlation IDs" description="Use Correlation IDs" category="Configuration" difficulty="INTERMEDIATE" tags=["Operations", "Deployment", "Correlation", "IDs"] unverified="user-authored ASP.NET correlation-ID middleware; not asserted by a mapped unit test"}
public class CorrelationIdMiddleware {
  private readonly RequestDelegate _next;

  public async Task InvokeAsync(HttpContext context) {
    var correlationId = context.Request.Headers["X-Correlation-ID"].FirstOrDefault()
      ?? TrackedGuid.NewMedo().ToString();  // UUIDv7 - Whizbang's id convention

    context.Response.Headers.Add("X-Correlation-ID", correlationId);

    using (_logger.BeginScope(new Dictionary<string, object> {
      ["CorrelationId"] = correlationId
    })) {
      await _next(context);
    }
  }
}
```

**Application Insights query**:

```kusto
traces
| where customDimensions.CorrelationId == "abc-123"
| order by timestamp asc
```

Inside the message pipeline you don't need middleware for this - every Whizbang `MessageEnvelope` already carries `CorrelationId` and `CausationId` in its hop chain, propagated automatically across outbox, transport, and inbox. The middleware above only bridges the HTTP edge into that flow.

### Custom Health Check

```csharp{title="Custom Health Check" description="Custom Health Check" category="Configuration" difficulty="ADVANCED" tags=["Operations", "Deployment", "Custom", "Health"] unverified="user-authored IHealthCheck example (OutboxHealthCheck); the framework SubscriptionHealthCheck is verified separately by SubscriptionHealthCheckTests"}
public class OutboxHealthCheck : IHealthCheck {
  public async Task<HealthCheckResult> CheckHealthAsync(
    HealthCheckContext context,
    CancellationToken ct = default
  ) {
    var backlog = await _db.ExecuteScalarAsync<int>(
      "SELECT COUNT(*) FROM wh_outbox WHERE processed_at IS NULL"
    );

    if (backlog > 10000) {
      return HealthCheckResult.Degraded(
        $"Outbox backlog is {backlog} messages",
        data: new Dictionary<string, object> { ["backlog"] = backlog }
      );
    }

    return HealthCheckResult.Healthy();
  }
}

## Key Takeaways

✅ **No Handler Found** - Call the generated `AddWhizbangDispatcher()` + `AddReceptors()` extensions
✅ **Message Not Published** - Check `AddWhizbang()` worker wiring, `wh_outbox` backlog, transport health checks
✅ **Duplicate Processing** - Inbox dedup is built in; make retry-sensitive receptors idempotent (`AttemptNumber`, `[ReceptorIdempotent]`)
✅ **High Latency** - Add database indexes, avoid N+1 queries
✅ **Memory Leak** - Use `await using` for connections, unsubscribe event handlers
✅ **Diagnostic Tools** - dotnet-counters, dotnet-trace, dotnet-dump, kubectl logs

## Troubleshooting Checklist

- [ ] Check logs in Application Insights or Kubernetes
- [ ] Verify DI registrations (receptors, perspectives, policies)
- [ ] Check database connectivity and connection string
- [ ] Verify Service Bus connection and permissions
- [ ] Check outbox/inbox tables for backlog
- [ ] Review slow query log in PostgreSQL
- [ ] Monitor metrics (CPU, memory, request rate, error rate)
- [ ] Use correlation IDs to trace requests across services
- [ ] Profile with dotnet-trace or BenchmarkDotNet
- [ ] Check health endpoints for service status
